# copy-feature

외부에서 본 좋은 기능/패턴/아이디어를 **그대로 복붙하는 대신**, atom-flow 하네스에 맞게 **재해석한 씨앗**으로 변환한다.

이 스킬은 분석가이자 번역가 역할만 한다. 실제 스킬 파일 생성·수정은 사용자 확인 후 다른 워크플로우(예: skill-creator)로 넘긴다.

## 입력

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `source_kind` | `enum<source-code \| spec-doc \| natural-language>` | ✅ | 입력 형태 (1단계 Intake에서 분기 기준) |
| `source_ref` | `string` | ✅ | URL·경로·자연어 아이디어 본문 |
| `target_scope` | `string` | ⬜ | 차용 범위(전체/일부/특정 부분). 생략 시 1단계에서 사용자에게 묻는다 |

## 출력

- 채팅 인라인 씨앗 문서 (🟢/🟡 케이스). 사용자가 명시 경로를 지정해야 파일로 저장.
- 🔴 케이스는 막힘 사유 + 해결 조건만 인라인 보고.

## 언제 쓰는가

- 다른 플러그인/프레임워크의 기능을 보고 "이거 우리도 갖고 싶다" 싶을 때
- 논문·블로그·트윗에서 본 아이디어를 atom-flow 방식으로 녹이고 싶을 때
- 기존 대형 스킬을 여러 개로 쪼개고 싶을 때
- 이식 여부를 사전에 판단해야 할 때 (노력 vs 이득)

## 언제 쓰지 않는가

- 이미 atom-flow 안에 있는 스킬을 수정/리팩터링: 이건 일반 편집 작업
- 완전히 새로운 스킬을 백지에서 설계: `skill-creator` 또는 브레인스토밍으로
- 단순히 "이거 어떻게 동작해?" 같은 설명 요청: 그냥 설명하면 됨

## 핵심 원칙

이 스킬이 복붙과 다른 이유는 다음 네 가지다.

**1. 원본을 존중하고 최소한만 수정한다.** ← 가장 중요
대부분의 오픈소스 스킬은 나보다 더 많이 고민하고 더 많은 사용자 피드백을 소화한 사람이 다듬은 결과물이다. 그러므로 기본 가정은 **"원본 그대로 가져온다"**다. 수정은 "이렇게 하면 더 좋을 것 같아서"가 아니라 **"atom-flow 하네스에서 그대로는 동작하지 않아서"** 일 때만 한다. 수정할 때도 범위를 최소로 잡는다 — 경로 변경, 이름 조정, 필수 의존성 대체 정도에서 그친다. 과감한 재구조화는 기본 경로가 아니다.

**2. 재해석은 최후의 수단이다.**
그대로 옮기는 게 진짜 불가능하다는 게 확인된 뒤에만 원본을 네 줄(문제/전략/메커니즘/경계)로 환원해 재설계한다. "이 원본의 본질만 뽑아 새로 쓰면 더 깔끔해질 텐데"라는 유혹은 대부분 과신이다. 원본이 그 모양인 데는 보통 이유가 있다.

**3. 적합성을 먼저 판정한다.**
멋있어 보인다고 다 맞지는 않는다. 원본이 대규모 상태 머신이나 멀티 프로세스 오케스트레이션에 의존한다면 그대로 가져오는 건 무리다. 판정은 3단계: **녹색(가능) / 황색(부분) / 적색(불가)**. 적색이어도 "왜 안 되는지"를 남겨두면 다음에 시도할 때 시간을 아낀다.

**4. 파일은 건드리지 않는다.**
이 스킬은 분석과 제안에서 끝난다. 실제 파일이 생기는 건 사용자가 "이대로 만들어줘" 하고 확인한 뒤다. 이렇게 나누면 (a) 아이디어를 빠르게 폐기·수정할 수 있고, (b) 원본에 대한 오해를 발견해도 비용이 작다.

## 작업 흐름

### 1단계 — 입력 수집 (Intake)

사용자가 던진 게 무엇인지부터 구분한다.

| 형태 | 처리 |
|------|------|
| **소스코드/리포** | URL 또는 경로 확보. 전체를 읽지 말고 README, 핵심 스킬/모듈 하나, 진입점만 본다. |
| **스펙/README/문서** | 섹션 목차부터 훑고, "What/Why/How" 중 How는 건너뛰어도 된다. |
| **자연어 아이디어** | 사용자에게 2-3개 구체화 질문을 한다. 무엇이 해결되는가? 누가 언제 쓰나? 이미 비슷한 게 있는가? |

확인할 것:
- **출처**: 라이선스, 저자의 의도 (그대로 옮겨도 괜찮은가 / 컨셉만 차용인가)
- **범위**: 사용자가 원하는 게 전체인지 일부인지. 대개 **일부**다. 전체를 원한다면 다시 물어볼 가치가 있다.

### 2단계 — 그대로 이식 게이트 (Passthrough gate)

네 줄 요약으로 내려가기 전에 먼저 묻는다: **"있는 그대로 가져올 수 있는가?"**

체크리스트:
- [ ] 파일을 그대로 복사해도 동작하는가? (경로만 바꾸면 끝)
- [ ] 이름 몇 개만 atom-flow 명명 규칙에 맞추면 되는가?
- [ ] 필요한 외부 의존성이 atom-flow 환경에 이미 있거나 드롭-인 대체가 가능한가?
- [ ] 원본의 라이선스가 복사를 허용하는가?

**모두 "예"라면 3단계(개념 추출)를 건너뛴다.** 적합도는 🟢, 씨앗은 "어떤 파일을 어디서 어디로, 어떤 최소 조정과 함께 옮겨라" 수준의 이식 지시문이면 충분하다. 이 경로가 이 스킬의 **기본 경로**다 — 대부분의 좋은 오픈소스 스킬은 여기서 끝난다.

하나라도 "아니오"라면 3단계로 내려간다. 내려가기 전에 "정말 수정이 필요한가, 아니면 내가 원본을 아직 충분히 안 읽은 건가?"를 한 번 더 자문한다.

### 3단계 — 개념 추출 (Concept extraction, 필요할 때만)

2단계에서 "그대로는 안 된다"가 확정됐을 때만 이 단계로 내려온다. 원본을 다음 네 줄로 압축한다. 못 압축하면 아직 이해하지 못한 것이다.

```
문제   : 이 기능이 없으면 사용자가 겪는 고통은?
전략   : 그 고통을 어떤 방식으로 해소하는가? (한 문장)
메커니즘: 전략이 동작하려면 무엇이 필요한가? (전제 조건, 자원, 상태)
경계   : 어디서부터는 이 기능의 책임이 아닌가?
```

예시 — superpowers의 brainstorming 스킬을 대상으로:
```
문제   : 구현부터 뛰어드는 바람에 요구가 틀어졌음을 늦게 발견한다
전략   : 코드 전에 의도·요구·설계에 대한 합의를 짧게 만든다
메커니즘: 사용자 인터뷰 + 2-3안 제안 + 합의문 산출물
경계   : 구현 실행은 다른 스킬의 몫
```

이 네 줄이 씨앗의 척추다. 디렉토리 구조나 파일 이름은 아직 신경 쓰지 않는다.

### 4단계 — 적합도 판정 (Fit judgment)

2단계 게이트를 통과했다면 이 단계는 사실상 "🟢 그대로 이식"으로 이미 결정된 것이다. 이 단계가 본격적으로 필요한 건 게이트에서 걸린 케이스뿐이다. 원본을 atom-flow에 녹였을 때 부딪히는 마찰을 체크리스트로 훑는다.

- [ ] **독립성**: 외부 서비스, 긴 러닝 프로세스, 공유 상태에 의존하는가?
- [ ] **입출력 명확성**: 입력과 출력이 명시적인가? 암묵적인 세션 상태에 의존하면 위험.
- [ ] **도구 요구**: atom-flow에서 이미 쓰는 도구만으로 가능한가? 새로 붙어야 하는 게 있으면 비용 증가.
- [ ] **라이선스/윤리**: 복사가 아닌 차용이 허용되는가?

판정:

**🟢 녹색 — 이식 가능**
두 가지 하위 케이스가 있다. 구분해서 명시한다.
- **🟢 그대로 이식**: 2단계 게이트를 통과한 경우. 파일 복사 + 경로/이름 최소 조정. 이 스킬이 추구하는 기본 형태다.
- **🟢 최소 수정 이식**: 체크리스트가 대부분 통과하지만 1-2개 항목에서 작은 조정이 필요한 경우. 조정 항목을 명시한다.

**🟡 황색 — 부분 이식**
일부는 그대로 되지만 어딘가 다르게 써야 한다. 어떤 부분이 조정되는지, 그리고 왜 조정이 **불가피한지**(선호가 아니라) 명시한다.
- 예: "원본은 5개 단계지만 atom-flow에서는 3단계로 통합. 2·3단계는 묶고 4단계는 별도 스킬로 분리 제안"
- 조정 근거가 "이렇게 하면 더 깔끔할 것 같아서"라면 재검토한다 — 원칙 1번 위반이다.

**🔴 적색 — 이식 불가**
체크리스트에서 핵심 항목이 걸린다. **왜** 걸리는지와 **어떻게 하면** 가능해질지(가설이라도)를 남긴다.
- 예: "원본은 사용자 세션 간 장기 메모리를 전제하는데 atom-flow는 현재 그런 저장소가 없음. 메모리 스킬이 생기면 녹색으로 전환 가능"

적색은 실패가 아니다. 적색 판정과 사유를 남기는 것 자체가 이 스킬의 기여다.

### 5단계 — 씨앗 합성 (Seed synthesis)

녹색/황색인 경우 씨앗 문서를 생성한다. 씨앗은 **아직 스킬이 아니다** — 사용자가 "이대로 만들자"고 동의한 뒤 skill-creator 같은 다른 스킬이 실제 파일을 짠다.

**🟢 그대로 이식 케이스의 씨앗은 짧다.** 아래 풀-포맷이 아니라 "원본 경로 → 목표 경로 + 바꿔야 할 최소 항목 N개"만 명시해도 충분하다. 이 스킬은 과도한 문서화를 요구하지 않는다.

씨앗 형식:

```markdown
# 씨앗: <이름 초안>

## 출처
- 원본: <링크/경로>
- 라이선스/이식 근거: <한 줄>

## 네 줄 요약
(2단계에서 뽑은 네 줄)

## 적합도
- 판정: 🟢/🟡/🔴
- 이유: <한두 문장>
- 황색/적색일 경우 조정안:
  - <bullet>

## 제안 형태
- 단일 스킬 / 쪼개진 N개 스킬 / 기존 스킬 확장 중 택1
- 각 스킬에 대해:
  - name (초안)
  - description (트리거 문구 3-5개 포함)
  - 본문 스케치 (한두 문단 또는 bullets)

## 원본에서 버리는 것
- <원본에서 의도적으로 옮기지 않는 부분과 이유>

## 확인이 필요한 결정
- <사용자가 골라줘야 할 포인트 2-3개>
```

씨앗은 **채팅 안에 표시**한다. 사용자가 저장 경로를 지정하지 않는 한 파일로 떨어뜨리지 않는다.

## 디렉토리 분류 가이드

이 스킬로 이식을 결정한 새 스킬은 atom-flow의 2-tier 구조에 배치한다.

| 디렉토리 | 역할 |
|---|---|
| `skills/` | 워크플로우 핵심 스킬 (brainstorming·implementing·writing-report 등) |
| `shared-skills/` | 보조성·범용 스킬 (여러 스킬에서 단발 호출되는 것) |

## 안티패턴

- **게이트를 건너뛰고 재해석부터 시작하기**: 2단계 게이트를 생략하면 필요 없는 재해석을 한다. "이거 내 스타일로 새로 쓰면 더 깔끔할 텐데"는 거의 항상 착각이다 — 원본이 그 모양인 데는 보통 이유가 있고, 나는 그 이유를 모를 가능성이 높다.
- **"내 취향대로" 재설계하기**: 수정 근거는 **하네스 제약**(이대로는 동작 안 함)이어야 하지 **선호**(이렇게가 더 깔끔)가 아니다. 선호는 원칙 1번을 위반하는 가장 흔한 방식이다.
- **원본을 통째로 읽고 요약하기**: 토큰만 쓰고 씨앗이 안 나온다. 먼저 게이트 체크, 막히면 네 줄 요약, 더 막히는 지점에서만 원본을 깊게 파고든다.
- **적합도 판정을 건너뛰고 바로 구현 제안하기**: 황색/적색 신호를 놓치면 2주 뒤 롤백한다.
- **파일을 먼저 만들고 물어보기**: 씨앗 단계에서는 쓰기 도구를 쓰지 않는다. 사용자가 "이대로 만들자"고 해야 다음 단계다.
- **"그냥 멋있어서" 가져오기**: "이 기능이 없으면 우리가 겪을 고통"을 한 줄로 못 쓰면 멈춘다.

## 산출물 체크리스트

완료 시 사용자에게 다음이 남아 있어야 한다.

- 🟢(그대로/최소 수정) / 🟡(부분) / 🔴(불가) 판정과 한 줄 사유
- (🟢 그대로 이식인 경우) 이식 지시문: 원본 경로 → 목표 경로, 바꿀 최소 항목 목록
- (🟢 최소 수정 / 🟡인 경우) 씨앗 문서 1개 + **네 줄 요약**
- (🔴인 경우) 막히는 지점과 해결 조건(가설이라도)
- 다음 액션에 대한 질문 1-3개 ("이대로 skill-creator로 넘길까요?", "조정 범위가 이 정도면 충분한가요?")

🟢 그대로 이식 케이스는 산출물이 짧을수록 좋다 — 불필요한 문서화는 원본 신뢰 원칙에 반한다.

## Terminal state

분석·제안 완료. 파일 수정은 사용자가 "이대로 만들자"고 확인한 뒤 다른 워크플로우(예: skill-creator, brainstorming)로 인계.

**Terminal output schema** (using-atom-flow §6.5.2 인용):
`{ status: success | warning, summary: "<🟢/🟡/🔴> verdict for <source>", next: user, artifacts: [<seed-doc path>] }`

- 씨앗 문서를 인라인으로만 보여줄 때는 `artifacts: []`. 사용자가 경로를 지정해 저장한 경우만 path 추가.
- 🔴(이식 불가)는 `status: warning` — 분석은 성공이지만 권장 임계 미달.
- 사용자에게 제시할 후속 후보(`skill-creator`·`brainstorming`·`drop` 등)는 본문에서 제안한다. `next`는 항상 `user`로 둬 enum 일관성을 유지한다.
